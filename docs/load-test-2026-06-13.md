# Load Test: 8-way Concurrent Draft Fan-out (2026-06-13)

First concurrent load test of the `vivijure-backend` RunPod serverless endpoint on `backend-v0.2.6`.

## Verdict: PASS

8 simultaneous draft renders, **8/8 completed, 0 failures**, scaled to 10 workers, **no throttling**. The one material finding is **cold-start tail latency**: copying model weights from R2 onto cold workers dominates wall-clock, and one weak node gated the whole batch by ~18 minutes.

## Method

- **Endpoint:** `t9wcvlxh8rc5la`, image `ghcr.io/skyphusion-labs/vivijure-backend:0.2.6` (Release #149).
- **Workload:** 8 async jobs via `POST /v2/{id}/run`, fanned out concurrently. Payload per job: `action=render`, `bundle_key=bundles/Packet_Chase.tar.gz` (6-shot project), `quality_tier=draft`, and a **unique project slug** (`loadtest-00`..`loadtest-07`) so v0.2.6's hash-gated keyframe cache cannot let later jobs reuse keyframes -- every render does full work.
- **Observation:** fleet-level via `/health` polling (queue depth, worker counts, per-job status); pod-level via SSH into all 8 live workers (`nvidia-smi`, VRAM, CPU).

## Results

| Metric | Result |
|---|---|
| Jobs completed | **8 / 8** |
| Failures | **0** |
| Workers scaled to | **10** (H200 x9 + 1 RTX PRO 6000 Blackwell) |
| Throttled / unhealthy | 0 / 0 (despite a "primary GPU supply low" banner) |
| VRAM used | 10-25 GB of 143 GB (H200) -- ample headroom, no OOM risk |
| Output | each job wrote a 6-shot `full.mp4` (4.7-5.8 MB) + 6 keyframes to R2 |

**Completion spread (from R2 artifact timestamps):**
- Bulk of the pack: **13:13:38 - 13:16:27** (within ~3 min of each other).
- One mid-straggler (`loadtest-02`): 13:25:13.
- **`loadtest-07` (the Blackwell) last at 13:34:31 -- ~18 min behind the pack.**

## The bottleneck: cold-start R2 weight-copy

Pod-level data during the run was the smoking gun: most workers sat at **0% GPU util with 100-179% CPU** for the bulk of their runtime. They were not rendering -- they were CPU/network-bound **staging model weights from R2**. The draft GPU work (few-step keyframes + Wan-Lightning 4-step i2v) is trivial; the cost is *getting the models onto a cold worker*. With 8 cold workers pulling from R2 at once, the staging contends and crawls.

Worst case was the lone **RTX PRO 6000 Blackwell**: it spent ~35 minutes copying weights before rendering a single frame (confirmed by direct observation -- "it finally finished copying the weights"). In a heterogeneous serverless pool, **tail latency is set by the weakest node that happens to land**, and RunPod assigns one at random. The Blackwell was worst on every axis: slower compute, lower VRAM ceiling (97 vs 143 GB), and (inferred from staging time) a weaker network link.

## Recommendation

Eliminate the cold-start weight-copy so a straggler can never gate a batch (tracked in the linked issue):
- **Pre-warm** a pool of hot workers, and/or
- **Bake weights into the image** instead of R2-mirror on cold start, and/or
- **Throttle/stage** concurrent cold-starts to avoid R2 egress contention.

Warm workers are fast; the entire pain is the cold fan-out.

## Bonus: what it validated

- **v0.2.6 multi-char fix generalizes.** `Packet_Chase`'s dual-character shots render cleanly separated (no bleed, no dark-blob) -- independent confirmation of the `lora_scale_per_slot=0.7` fix on a *different* project than NEON HALFLIFE.
- **Draft-tier quality is strong** (and it is the floor, not the ceiling): clean keyframes, consistent cast identity across shots, with zero finish passes applied.
